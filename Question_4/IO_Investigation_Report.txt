=== File Access and I/O Investigation Report ===

1. Open Files Identified
Started a background process (sleep 100, PID 473) and inspected its open
files using /proc/473/fd and lsof -p 473. The process had file descriptors
0 (stdin), 1 (stdout), 2 (stderr) — all connected to the terminal device
/dev/pts/0. lsof additionally showed the executable file (txt), working
directory (cwd), root directory (rtd), and shared libraries (mem) loaded
by the process, such as libc.so.6 and ld-linux-x86-64.so.2.

2. File Descriptor Observations
Every Linux process is given three standard file descriptors by default:
0 = stdin, 1 = stdout, 2 = stderr. /proc/<pid>/fd exposes these (and any
additional open files) as symlinks, which is how tools like lsof gather
their information. Processes can open additional file descriptors as
needed (numbered sequentially from 3 onward).

3. Output and Error Redirection Results
- ls /etc > output.txt 2> error.txt: succeeded, so output.txt contained
  the full directory listing and error.txt was empty.
- ls /nonexistent_folder > output2.txt 2> error2.txt: failed, so
  output2.txt was empty and error2.txt contained:
  "ls: cannot access '/nonexistent_folder': No such file or directory"
- echo "combined output" > combined.txt 2>&1: the 2>&1 syntax redirected
  stderr to the same destination as stdout (fd 1), merging both streams
  into combined.txt, which correctly showed "combined output".

4. Resource Limits Observed
- open files (-n): 10240 (max file descriptors this shell/process may hold open)
- max user processes (-u): 30907
- pending signals (-i): 30907
- stack size (-s): 8192 KB
- core file size (-c): 0 (core dumps disabled)
- Most other limits (cpu time, virtual memory, file size, etc.) were unlimited.

5. Explanation of How Linux Manages File I/O
Linux follows the principle that "everything is a file" — regular files,
devices, pipes, and sockets are all accessed through the same file
descriptor interface. Every process receives fds 0/1/2 by default and can
open more as needed, tracked under /proc/<pid>/fd. Shell redirection
operators (>, 2>, 2>&1) let the shell rewire these descriptors before a
program runs, without the program itself needing any special code to
support it. ulimit enforces per-process ceilings (like max open files) at
the kernel level, preventing a single process from exhausting system
resources.
